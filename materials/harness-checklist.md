# Harness Engineering 체크리스트

> AI가 잘 일하는 환경을 설계하는 기술 — 수강생 배포용

## 3단계 성숙도

| 레벨 | 이름 | 비유 |
|------|------|------|
| **L1** | 시작하기 | 교과서로 문법 배우기 |
| **L2** | 내 것으로 만들기 | 교과서 문법으로 내 글 쓰기 |
| **L3** | 자율 운영 | 글쓰기 스타일이 자리잡음 |

---

## 축 1: 준비 (Scaffolding)

> 자가진단: "AI가 내 프로젝트에 처음 들어왔을 때, 뭘 해야 하는지 스스로 파악할 수 있는가?"

| # | 체크 항목 | L |
|---|----------|:-:|
| 1 | 디렉토리 구조만 보고 프로젝트의 역할 구분을 알 수 있다 | L1 |
| 2 | AI에게 프로젝트 설명을 전달하는 파일이 있다 (CLAUDE.md 등) | L1 |
| 3 | 긴 맥락은 별도 문서로 분리하고, 메인 파일에서 참조한다 | L1 |
| 4 | 민감 파일(.env, 시크릿 등)을 AI가 함부로 수정하지 못하게 보호하고 있다 | L1 |
| 5 | 도메인별 규칙 파일이 분리되어 있고, 관련 작업 시에만 자동 로드된다 | L2 |
| 6 | 프로젝트 전용 자동화(Skill, Hook 등)가 1개 이상 있다 | L2 |
| 7 | AI 행동의 범위를 명시적으로 제한하고 있다 (샌드박스, 파일 경로 제한, 권한 설정 등) | L2 |
| 8 | 새 팀원(사람이든 AI든)이 설정 파일만 읽고 개발을 시작할 수 있다 | L3 |

**Before → After**: 매번 "이 프로젝트는 모노레포고 서버는 NestJS이고..." 설명 → AI가 첫 턴부터 구조를 파악하고 바로 작업 시작

---

## 축 2: 맥락 (Context)

> 자가진단: "AI에게 같은 말을 반복하고 있지 않은가?"

| # | 체크 항목 | L |
|---|----------|:-:|
| 1 | 반복해서 지시하는 내용을 설정 파일에 적어놨다 | L1 |
| 2 | 설정 파일이 간결하다 (핵심만, 길어지면 분리) | L1 |
| 3 | AI가 예상과 다르게 행동하면, "내가 당연하다고 생각한 것"을 찾아 기록한다 (암묵지 → 명시지) | L1 |
| 4 | 정보의 범위를 구분해서 배치했다 (내 전체 설정 / 프로젝트 설정 / 폴더 설정) | L2 |
| 5 | 도메인 용어, 핵심 정책, 과거 장애 기록 등이 AI가 참조할 수 있는 형태로 문서화되어 있다 | L2 |
| 6 | 외부 시스템(이슈 트래커, 메신저, DB 등)을 AI가 직접 조회할 수 있게 연결했다 | L2 |
| 7 | 필요한 맥락이 "필요할 때만" 로드된다 — 한꺼번에 다 주지 않는다 (Progressive Disclosure) | L3 |

**Before → After**: "당연히 교육용이니까 코드가 짧아야지" → AI는 300줄짜리 코드를 생성 / 명시 후 → AI가 10줄로 간결하게 작성

> 핵심: "AI한테 안 말했는데 왜 모르지?" → 그게 암묵지. 발견할 때마다 기록하면 하네스가 진화한다.

## 축 3: 실행 (Orchestration)

> 자가진단: "AI에게 '해줘' 한 마디로 시키고 있는가, 아니면 일하는 방식을 설계해주고 있는가?"

| # | 체크 항목 | L |
|---|----------|:-:|
| 1 | "XXX 해줘" 대신 **배경, 목적, 제약**을 함께 전달한다 | L1 |
| 2 | 위임할 때 **완료 기준**(어떤 상태가 되면 "끝"인지)을 명시한다 | L1 |
| 3 | 복잡한 작업 전에 **계획을 먼저 세우게** 한다 (바로 실행하지 않게) | L2 |
| 4 | 작업 규모에 따라 **혼자 시키기 / 부하 파견 / 팀 구성**을 구분해서 쓴다 | L2 |
| 5 | 장시간 작업 시 **세션 인수인계 문서**를 남긴다 — 다음 세션이 "어디까지 했고, 뭐가 됐고, 뭐가 안 됐는지" 이어받을 수 있다 | L2 |
| 6 | 완료 기준 충족까지 **AI가 스스로 반복 시도**하는 구조를 활용한다 | L3 |
| 7 | 커스텀 계획 프로세스가 있다 (인터뷰 → 스펙 생성 → 실행) | L3 |
| 8 | 독립적인 작업을 **병렬로 동시 실행**하는 구조가 있다 | L3 |

**Before → After**: 긴 작업 중 세션이 끊기면 처음부터 다시 설명 → 인수인계 문서가 남아서 다음 세션이 바로 이어받기

> 참고 데이터: 계획 없이 실행 $9/20분(깨짐) vs 계획 후 실행 $125/3시간50분(완성)

## 축 4: 검증 (Verification)

> 자가진단: "AI 결과물을 눈으로만 확인하고 있는가, 구조적으로 검증하는가?"

| # | 체크 항목 | L |
|---|----------|:-:|
| 1 | AI가 작업 후 **테스트를 돌릴 수 있는 환경**이 있다 | L1 |
| 2 | **코드 포맷터/린터**가 AI 작업 후 자동으로 적용된다 | L1 |
| 3 | 위험한 작업(삭제, 배포, 민감파일 수정 등)을 **자동으로 차단**하는 장치가 있다 | L1 |
| 4 | 실수해도 되돌릴 수 있는 구조다 (브랜치 분리, 격리 환경) | L2 |
| 5 | **시뮬레이션(Dry-run) → 확인 → 실행** 흐름이 위험 작업에 적용되어 있다 | L2 |
| 6 | AI가 실수했을 때 **자동으로 감지하고 재시도하는 복구 구조**가 있다 (에러 리커버리, 재시도 루프) | L2 |
| 7 | 격리된 환경에서 전체 흐름(E2E)을 검증할 수 있다 | L2 |
| 8 | **만드는 AI와 검증하는 AI가 분리**되어 있다 (같은 맥락에서 자기 검증 X) | L3 |

**Before → After**: AI가 에러를 내면 사람이 직접 디버깅 → 에러 감지 시 자동 재시도 + 3회 실패하면 사람에게 보고

> 핵심: "동작한다"는 것과 "안전하다"는 것은 다르다. 눈으로 확인하는 것의 한계를 인식하라.

## 축 5: 개선 (Compounding)

> 자가진단: "내 Harness가 시간이 갈수록 나아지고 있는가, 복잡해지고 있는가?"

| # | 체크 항목 | L |
|---|----------|:-:|
| 1 | AI가 반복 실수하는 패턴을 발견하면 **규칙으로 만든다** | L1 |
| 2 | 내가 3번 이상 반복하는 작업을 인지하고 있다 | L1 |
| 3 | 반복 작업을 **자동화(Skill 등)로 만들었다** | L2 |
| 4 | AI 작업을 **관찰/추적**할 수 있다 (토큰 사용량, 자동화 사용 빈도, 세션 로그 분석) | L2 |
| 5 | 안 쓰는 자동화, 규칙, 연결을 정기적으로 **정리한다** — 쌓이면 AI도 혼란 | L3 |

**Before → After**: AI가 매번 같은 실수를 반복 → 규칙 추가 후 같은 실수가 사라짐

> 원칙: 3번 반복하면 자동화, 3번 틀리면 규칙, 안 쓰는 건 치운다.
>
> "51번째 작업의 출발점은 50번째보다 월등하다." — Compound, don't compact.

## "잘 가고 있다"는 신호

| 신호 | 의미 |
|------|------|
| 같은 말을 두 번 하지 않는다 | 맥락 전달이 잘 되고 있다 |
| 실수가 규칙이 된다 | 개선 루프가 돌고 있다 |
| 차단 장치가 뭔가를 막고 있다 | 사고가 구조적으로 예방되고 있다 |
| 새 세션에서 설명 시간이 줄어든다 | 설정이 잘 작동하고 있다 |
| AI가 건드리면 안 되는 것을 안 건드린다 | 경계 설정이 효과적이다 |
| 불필요한 것이 계속 줄어든다 | 복잡해지는 게 아니라 단순해지고 있다 |

---

## "실패하고 있다"는 징후

| 징후 | 의미 |
|------|------|
| 설정 파일이 점점 길어지기만 한다 | 분리/정리 없이 쌓이고 있다 |
| AI에게 시킨 것의 절반을 다시 고친다 | 맥락 전달 또는 검증이 부족하다 |
| "동작하니까 됐지"로 넘어간다 | 검증 루프가 없다 (vibe coding 함정) |
| 만들어놓은 자동화를 정작 안 쓴다 | 필요 없는 것을 만들었거나 접근성이 낮다 |
| 세션마다 처음부터 다시 설명한다 | 맥락 축적이 안 되고 있다 |
| 에러가 나면 사람이 직접 고친다 | 에러 복구 구조가 없다 |
